5. 知识总结

5.1 SSH 认证方式对比

认证方式 原理 安全性 适用场景 配置要点
密码认证 用户输入密码，服务器验证 /etc/shadow 较低（易被暴力破解） 临时访问、首次连接 PasswordAuthentication yes
密钥认证 客户端用私钥签名，服务器用公钥验证 高（无需传输密码） 日常管理、自动化脚本 生成密钥对，公钥放入 ~/.ssh/authorized_keys，PubkeyAuthentication yes

5.2 SSH 密钥对管理要点

  • 私钥（id_ed25519）严格保密，权限应为 600，不可泄露。
  • 公钥（id_ed25519.pub）可公开，添加到远程主机的 authorized_keys 文件中。
  • authorized_keys 文件权限必须为 600，.ssh 目录权限为 700，否则 SSH 会拒绝使用。
  • 可使用 ssh-copy-id 自动复制公钥，避免手动操作错误。
  • 推荐使用 ED25519 算法（-t ed25519），比 RSA 更短更安全。

5.3 SCP vs rsync 对比

工具 特点 优缺点 适用场景
SCP 基于 SSH，加密传输；每次全量复制 简单易用，但不支持增量；传输大目录重复数据多 少量文件一次性传输
rsync 增量传输（只传差异）；支持压缩、删除、过滤 效率高，支持断点续传（需结合–partial）；命令复杂 目录同步、定期备份、持续部署

rsync 常用参数：

参数 作用
-a 归档模式（保留权限、时间戳、递归等）
-v 详细输出
-z 传输时压缩
-e ssh 指定使用 SSH 作为传输通道
--delete 删除目标端多余文件（保持完全一致）
--exclude 排除指定文件/目录
--partial 保留部分传输的文件，便于断点续传

5.4 SSH 安全加固建议

  • 禁用 root 登录：PermitRootLogin no
  • 更改默认端口（如 2222），降低被扫描风险
  • 仅允许密钥登录（PasswordAuthentication no），但本次实验保留密码以便演示
  • 使用 fail2ban 防止暴力破解
  • 定期更新 OpenSSH 版本

5.5 常用 SSH 命令速查

命令 说明
ssh user@host 远程登录
ssh user@host "command" 远程执行单条命令
ssh-copy-id user@host 复制公钥到远程
scp file user@host:/path/ 上传文件
scp -r dir user@host:/path/ 上传目录
scp user@host:/path/file ./ 下载文件
rsync -avz -e ssh src/ user@host:/dst/ 远程增量同步推送
rsync -avz -e ssh user@host:/src/ ./dst/ 远程增量同步拉取

6. 出现问题及解决方案

问题现象 可能原因 解决方法 是否解决
执行 ssh-copy-id 提示 Permission denied 远程用户密码错误或 SSH 服务未允许密码认证 确认密码正确；检查远程 /etc/ssh/sshd_config 中 PasswordAuthentication yes 并重启服务 ✅
免密登录仍提示输入密码 公钥未正确添加，或 authorized_keys 权限不对 检查 ~/.ssh/authorized_keys 内容是否正确；执行 chmod 600 ~/.ssh/authorized_keys；检查 .ssh 目录权限为 700 ✅
SCP 传输目录时提示 not a regular file 未使用 -r 参数 使用 scp -r 递归传输目录 ✅
rsync 远程同步失败，提示 command not found 远程主机未安装 rsync 在远程主机执行 sudo apt install rsync -y ✅
rsync 同步后文件权限/时间戳变化 未使用 -a 归档选项 使用 rsync -avz 保留属性和时间 ✅
使用 ssh node2 提示 Could not resolve hostname 未配置 /etc/hosts 或 ~/.ssh/config 在 hosts 中添加解析（如 192.168.56.103 node2），或正确配置 config 文件 ✅
生成密钥时设置密码短语，导致免密失效 密钥加密需要输入密码 重新生成密钥，不设置密码短语（直接回车）；若已生成可执行 ssh-add 添加私钥到代理 ✅
公钥复制到多台机器后，某台仍要求密码 目标机器的 ~/.ssh/authorized_keys 末尾缺少换行 确保公钥内容单独一行，且末尾有换行符 ✅

7. 心得体会

技术层面

本次实验让我深入理解了 SSH 协议在远程管理中的核心地位。通过配置密钥认证，我认识到了公钥/私钥体制的便利性与安全性——一次配置，后续所有操作（登录、SCP、rsync）均可免密进行，极大提升了工作效率。实验过程中，我掌握了 ssh-copy-id 的便捷用法，理解了 authorized_keys 文件的权限要求，这些细节在实际生产环境中至关重要。

SCP 和 rsync 的对比让我认识到，增量同步在数据量较大时具有明显的性能优势。rsync 的 --delete 参数能够保持两端完全一致，非常适合用于备份和部署场景。同时，我也体会到命令行工具的灵活性——通过参数组合可以满足各种复杂需求（如排除特定文件、压缩传输等）。

此外，SSH 配置文件 ~/.ssh/config 的别名功能极大简化了日常操作，将冗长的 ssh dev@192.168.56.103 简化为 ssh node2，体现了 Linux 工具追求高效的设计理念。

思维层面

SSH 作为开源网络协议，其广泛应用体现了互联网协作的安全基石。在当今网络安全形势日益严峻的背景下，掌握加密通信和身份认证技术是每位开发运维人员的必备素养。实验中我们禁止 root 登录、使用密钥认证，这些小细节体现了 最小权限原则 和 纵深防御 的安全思想。

同时，rsync 和 SCP 等工具的开源属性，使得开发者能够自由使用并贡献改进，这种开放共享的精神推动了云计算和自动化运维的高速发展。应理解其背后的安全模型和设计哲学，在实际工作中做到 便捷与安全并重，为构建稳定可靠的信息系统贡献力量。
