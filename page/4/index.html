五、知识总结

1、静态站点（SSG）vs 动态站点

对比维度 静态站点（SSG） 动态站点（如 WordPress）
页面生成时机 构建时预先生成所有页面 用户访问时由服务器实时生成
数据库 不需要 通常需要
访问速度 快（纯静态文件） 较慢（需执行后端代码+查数据库）
安全性 高（无后端攻击面） 需额外防护
部署方式 上传静态文件即可 需配置服务器环境和数据库
适用场景 博客、文档站、个人网站 电商、社交、CMS

2、Hexo 部署工作流

编写 Markdown → hexo generate → 生成 public/ 静态文件 → hexo deploy → 推送到 GitHub Pages → 全球访问

六、出现问题

问题现象 可能原因 解决方法
hexo init 速度极慢或失败 网络无法访问 GitHub 使用 Gitee 镜像源
hexo deploy 提示 Repository not found 仓库地址配置错误或仓库未创建 检查仓库名称是否与 GitHub 创建的仓库名完全一致
部署时提示 Permission denied SSH 公钥未添加到 GitHub 账户 将公钥添加到 GitHub Settings → SSH and GPG keys
部署后 GitHub Pages 显示 404 仓库名称不是 用户名.github.io 检查仓库命名是否正确，确保为公开仓库
GitHub Actions 失败 Token 缺少 workflow 权限 重新生成 Token，同时勾选 repo 和 workflow 权限
博客图片无法显示 post_asset_folder没有true，或者图片路径不对，又或者缺少marked 修改文件路径，引入marked的俩个配置，post_asset_folder为true

七、心得体会

通过本次实验，我完整经历了 Web 应用从开发到部署的全过程。Hexo 作为静态站点生成器，通过预编译 Markdown 文件生成纯静态 HTML，既保证了极快的访问速度，又消除了后端攻击的风险。

在部署环节，我配置了 SSH 密钥实现免密部署，体验了 GitHub Actions 自动化部署的便利。GitHub Pages 提供的免费托管服务，加上 Hexo 这样的开源工具，使得个人开发者能够零成本搭建和发布自己的网站，充分体现了开源社区的共享精神。同时，我也认识到国内虽然也有 Gitee 等代码托管平台，但在生态建设方面与 GitHub 仍有差距。
